Website Speed Test

A website speed test measures how quickly a server responds to an HTTP request. This is useful for checking your own server's performance or comparing response times across different hosting providers.

Key metrics

MetricWhat it measuresGood target
TTFBTime to First Byte — server processing + network latency< 200ms
Total timeFull response including download< 1s
Response sizeBytes transferredDepends on content
HTTP statusWhether the request succeeded200

TTFB (Time to First Byte) is the most important metric for server performance. It measures how long the server takes to begin responding — covering DNS resolution, TCP handshake, TLS negotiation, and server processing time.

From the command line

# Measure response time breakdown
curl -o /dev/null -s -w "\
DNS:        %{time_namelookup}s\n\
Connect:    %{time_connect}s\n\
TLS:        %{time_appconnect}s\n\
TTFB:       %{time_starttransfer}s\n\
Total:      %{time_total}s\n\
Size:       %{size_download} bytes\n" \
https://example.com

💡 Tip: High TTFB often means a slow database query or missing cache. High total time with low TTFB usually means a large response payload — optimize images and minify assets.

Port Checker

A port checker tests whether a specific TCP port on a remote host is open and accepting connections. This is essential for verifying firewall rules, checking if a service is running, or diagnosing why a connection is refused.

Common ports and their services

PortProtocolService
21TCPFTP
22TCPSSH
25TCPSMTP (email)
53TCP/UDPDNS
80TCPHTTP
443TCPHTTPS
3306TCPMySQL
5432TCPPostgreSQL
6379TCPRedis
27017TCPMongoDB

Port states explained

  • Open — a service is listening and accepted the connection
  • Closed — the host responded but no service is listening on that port
  • Filtered — no response (firewall is dropping packets silently)

From the command line

# Check a single port with netcat
nc -zv example.com 443

# Check multiple ports with nmap
nmap -p 22,80,443 example.com

# Quick TCP connection test with bash
timeout 2 bash -c 'echo >/dev/tcp/example.com/443' \
  && echo "open" || echo "closed/filtered"

⚠️ Note: Never port scan systems you don't own or have explicit permission to test. Unauthorized scanning may violate computer misuse laws.

Ping

Ping is the most basic network diagnostic tool. It sends ICMP Echo Request packets to a host and measures the round-trip time (RTT). Use it to verify a host is reachable and measure latency.

Reading ping output

PING google.com (142.250.74.46): 56 data bytes
64 bytes from 142.250.74.46: icmp_seq=0 ttl=118 time=12.4 ms
64 bytes from 142.250.74.46: icmp_seq=1 ttl=118 time=11.9 ms
64 bytes from 142.250.74.46: icmp_seq=2 ttl=118 time=12.1 ms
64 bytes from 142.250.74.46: icmp_seq=3 ttl=118 time=11.8 ms

--- google.com ping statistics ---
4 packets transmitted, 4 received, 0% packet loss
round-trip min/avg/max/stddev = 11.8/12.1/12.4/0.2 ms
FieldMeaning
icmp_seqPacket sequence number
ttlTime to live — hops remaining (starts at 64 or 128)
timeRound-trip time in milliseconds
packet loss% of packets that didn't return
stddevJitter — variation in latency (lower is better)

Latency benchmarks

RTTQualityTypical scenario
< 5msExcellentSame datacenter
5–20msVery goodSame city
20–50msGoodSame country
50–100msAcceptableCross-continent
100–200msNoticeableIntercontinental
> 200msPoorDistant servers or congestion

Common ping flags

# Linux/Mac: send 10 packets
ping -c 10 example.com

# Set packet size (test MTU issues)
ping -s 1400 example.com

# Flood ping (requires root) — stress test
ping -f example.com

# Windows equivalent
ping -n 10 example.com

💡 Tip: If ping shows 0% packet loss but your site feels slow, check the TTFB with a speed test — latency alone doesn't explain slow page loads. The bottleneck might be server-side processing.

Traceroute

Traceroute maps the path packets take across the network, showing each router (hop) between you and the destination. It's invaluable for finding where in the network a problem lies.

How traceroute works

Traceroute sends packets with increasing TTL values starting at 1. Each router decrements the TTL; when it reaches 0 the router sends back a "Time Exceeded" message. By incrementing the TTL, traceroute reveals each hop in the path.

Reading traceroute output

traceroute to google.com (142.250.74.46)
 1  192.168.1.1      1.2 ms   1.1 ms   1.0 ms   (your router)
 2  10.0.0.1         8.4 ms   8.6 ms   8.3 ms   (ISP gateway)
 3  72.14.215.165   11.2 ms  11.0 ms  11.3 ms   (ISP backbone)
 4  * * *                                        (no response)
 5  142.250.74.46   12.1 ms  12.0 ms  11.9 ms   (destination)

Each line shows: hop number, router IP/hostname, and three round-trip time measurements. Stars (* * *) mean the router didn't respond to traceroute — this is normal for many routers that block ICMP and doesn't necessarily indicate a problem.

What to look for

  • Sudden latency jump — a hop where RTT increases sharply is likely the bottleneck
  • All stars after a point — packets are being dropped beyond that hop
  • Routing loops — if you see the same IP multiple times, BGP routing may be misconfigured
  • Unexpected geography — traffic going through distant countries adds latency

Command line usage

# Linux
traceroute example.com

# Mac
traceroute example.com

# Windows
tracert example.com

# Use TCP instead of ICMP (bypasses some firewalls)
traceroute -T -p 443 example.com

# Limit hops
traceroute -m 20 example.com

Diagnosing Network Problems: A Workflow

When troubleshooting a connectivity issue, work through these steps in order:

  1. Ping the destination — Is the host reachable? What's the latency?
  2. Check the port — Is the specific service port open? (e.g., port 443 for HTTPS)
  3. Run traceroute — Where does the path slow down or fail?
  4. Test speed — What is the TTFB and total response time?
  5. Check DNS — Is the domain resolving to the right IP?

💡 Tip: Run the same tests from multiple locations (your machine + an online tool) to distinguish between a problem on your end vs. the remote server.

MTU and Packet Fragmentation

Maximum Transmission Unit (MTU) is the largest packet size a network can carry. Mismatched MTU settings cause intermittent failures that are hard to diagnose — large packets get silently dropped while small packets (like pings) succeed.

# Find the path MTU to a host
# Try progressively smaller sizes until it works
ping -M do -s 1472 example.com    # 1472 + 28 header = 1500 (standard Ethernet)
ping -M do -s 1400 example.com    # Works if 1472 fails (VPN or PPPoE)

# On Windows
ping -f -l 1472 example.com

If large pings fail but small ones succeed, set your router's MTU to a smaller value (1400 is safe for most VPN/PPPoE setups).
