What is a Unix Timestamp?

A Unix timestamp (also called epoch time or POSIX time) is the number of seconds that have elapsed since January 1, 1970 at 00:00:00 UTC. This point in time is known as the Unix Epoch.

For example, the timestamp 1700000000 corresponds to November 14, 2023 at 22:13:20 UTC. It is just a large integer — simple to store, compare, sort and transmit.

The Unix timestamp is used in virtually every computer system, programming language, database, and operating system on the planet. Knowing how to work with it is a fundamental developer skill.

Why January 1, 1970?

The choice of 1970 was somewhat arbitrary — it was a round number near the time when Unix was being developed at Bell Labs in the late 1960s and early 1970s. The engineers needed a fixed reference point and chose the start of 1970 as a convenient date. It has remained the universal standard ever since.

Seconds vs Milliseconds — The Most Common Mistake

Different systems use different resolutions:

  • Seconds — Unix/Linux, most server-side languages, databases
  • Milliseconds — JavaScript's Date.now(), Java's System.currentTimeMillis()
  • Microseconds — Python's time.time() returns a float with microsecond precision
  • Nanoseconds — Go's time.Now().UnixNano()

⚠️ Common bug: Passing a millisecond timestamp where seconds are expected (or vice versa) produces a date wildly off by 1000x. If you see a date in the year 47000, you are likely passing milliseconds where seconds are expected. If you see 1970-01-01, you are probably passing seconds where milliseconds are expected.

Working with Unix Timestamps in Code

JavaScript

// Current timestamp in seconds
Math.floor(Date.now() / 1000)

// Current timestamp in milliseconds
Date.now()

// Convert timestamp (seconds) to Date
new Date(1700000000 * 1000)

// Convert Date to timestamp (seconds)
Math.floor(new Date('2023-11-14').getTime() / 1000)

// Format a timestamp
new Intl.DateTimeFormat('en-US', {
  dateStyle: 'full',
  timeStyle: 'long',
  timeZone: 'UTC'
}).format(new Date(1700000000 * 1000))

Python

import time
from datetime import datetime, timezone

# Current timestamp (float, seconds with microseconds)
time.time()

# Current timestamp as integer
int(time.time())

# Convert timestamp to datetime
datetime.fromtimestamp(1700000000, tz=timezone.utc)

# Convert datetime to timestamp
datetime(2023, 11, 14, tzinfo=timezone.utc).timestamp()

# Format
dt = datetime.fromtimestamp(1700000000, tz=timezone.utc)
dt.strftime('%Y-%m-%d %H:%M:%S %Z')

Go

import "time"

// Current timestamp in seconds
time.Now().Unix()

// Current timestamp in milliseconds
time.Now().UnixMilli()

// Convert timestamp to time.Time
t := time.Unix(1700000000, 0).UTC()

// Format
t.Format(time.RFC3339)  // "2023-11-14T22:13:20Z"
t.Format("2006-01-02")  // "2023-11-14"

Bash / Linux

# Current timestamp
date +%s

# Convert timestamp to readable date
date -d @1700000000

# Convert readable date to timestamp
date -d "2023-11-14 22:13:20 UTC" +%s

# macOS (uses BSD date)
date -r 1700000000
date -j -f "%Y-%m-%d" "2023-11-14" +%s

SQL

-- MySQL / MariaDB
SELECT FROM_UNIXTIME(1700000000);
SELECT UNIX_TIMESTAMP('2023-11-14 22:13:20');
SELECT UNIX_TIMESTAMP(NOW());

-- PostgreSQL
SELECT TO_TIMESTAMP(1700000000);
SELECT EXTRACT(EPOCH FROM NOW())::BIGINT;
SELECT EXTRACT(EPOCH FROM TIMESTAMP '2023-11-14 22:13:20');

-- SQLite
SELECT datetime(1700000000, 'unixepoch');
SELECT strftime('%s', 'now');

Timezones and UTC

Unix timestamps are always in UTC. They have no timezone — a timestamp is the same number everywhere in the world. The timezone only matters when you convert a timestamp to a human-readable date for display.

💡 Best practice: Store and transmit timestamps as UTC Unix timestamps. Only convert to a local timezone at the very last moment — in the UI layer, when displaying to the user. This avoids an entire class of timezone-related bugs.

# Example: correct way to handle timezones
# 1. Store the event time as UTC timestamp
event_ts = 1700000000  # stored in DB as UTC

# 2. Convert to user's timezone only for display
from datetime import datetime
import pytz

user_tz = pytz.timezone('America/New_York')
dt = datetime.fromtimestamp(event_ts, tz=pytz.utc)
local = dt.astimezone(user_tz)
print(local.strftime('%Y-%m-%d %I:%M %p %Z'))
# Output: 2023-11-14 05:13 PM EST

The Year 2038 Problem (Y2K38)

On January 19, 2038 at 03:14:07 UTC, the Unix timestamp will overflow a 32-bit signed integer (max value: 2,147,483,647). Systems that store timestamps as 32-bit integers will roll over to a negative number, causing dates to be interpreted as 1901.

Most modern systems have already migrated to 64-bit timestamps, which won't overflow for approximately 292 billion years. However, legacy embedded systems, older databases, and some file systems may still be affected. Always use 64-bit integers for timestamps in new code.

ISO 8601 — The Other Universal Format

While Unix timestamps are great for machines, humans prefer readable dates. The international standard for date strings is ISO 8601, which looks like 2023-11-14T22:13:20Z. The Z at the end means UTC (Zulu time).

# ISO 8601 formats
2023-11-14                    # date only
2023-11-14T22:13:20Z          # datetime UTC
2023-11-14T22:13:20+05:30     # datetime with offset
2023-11-14T22:13:20.123Z      # with milliseconds

ISO 8601 strings are sortable alphabetically, unambiguous across locales, and universally supported. Use them in logs, APIs, and config files.

Frequently Asked Questions

What is epoch time?

Epoch time is another name for Unix timestamp — the number of seconds since January 1, 1970. The "epoch" is the reference point (the Unix Epoch).

Is Unix timestamp the same everywhere?

Yes. A Unix timestamp represents the same moment in time everywhere in the world. It is always UTC-based. The local time equivalent will vary by timezone, but the timestamp number is universal.

How do I get the current timestamp?

In most languages: JavaScript: Math.floor(Date.now()/1000), Python: int(time.time()), Go: time.Now().Unix(), Bash: date +%s, PHP: time().

Why does JavaScript use milliseconds?

JavaScript's Date object uses milliseconds for higher precision in interactive applications. This is a design decision from the early days of the language. Always divide by 1000 when converting to Unix seconds.
