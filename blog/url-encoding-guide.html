What is URL Encoding?

URL encoding (also called percent encoding) is a mechanism to represent special characters in a URL using only safe ASCII characters. A URL can only contain a limited set of characters — letters, digits, and a handful of special characters like -, _, ., and ~. Any other character must be encoded.

Encoding works by replacing the character with a percent sign % followed by two hexadecimal digits representing the character's ASCII (or UTF-8) byte value. A space, for example, becomes %20.

Why Does URL Encoding Exist?

URLs were defined by RFC 3986, which specifies that a URL is made up of a limited set of ASCII characters. The reason for this restriction is that URLs must be reliably transmitted over the internet, and older protocols and systems only support ASCII.

Additionally, certain characters have special meaning within URLs. For example, & separates query parameters and # marks a fragment. If your data contains these characters, they would be misinterpreted by the browser or server unless encoded.

Safe Characters That Don't Need Encoding

These characters are allowed in URLs without encoding (per RFC 3986 unreserved characters):

A-Z  a-z  0-9  -  _  .  ~

Additionally, some characters are "reserved" and only need encoding when they appear in a context where they would otherwise be misinterpreted (like / inside a query value).

The Anatomy of a URL

https://user:pass@example.com:8080/path/to/page?key=value&foo=bar#section
│       │    │    │           │    │              │              │
│       │    │    │           │    │              │              └ fragment
│       │    │    │           │    │              └ query string
│       │    │    │           │    └ path
│       │    │    │           └ port
│       │    │    └ host
│       │    └ password
│       └ username
└ scheme

Each component has its own encoding rules. For example, the path uses %2F to encode a literal slash that is part of a filename, while an actual path separator remains as /.

encodeURI vs encodeURIComponent

JavaScript has two encoding functions, and choosing the wrong one is a very common mistake:

encodeURI()

Encodes a complete URL. It leaves reserved characters like /, ?, &, =, # intact because they have meaning in the URL structure. Use this when you have a complete URL that just needs special characters in the path or hostname encoded.

encodeURI('https://example.com/search?q=hello world')
// → 'https://example.com/search?q=hello%20world'

encodeURI('https://example.com/path?a=1&b=2')
// → 'https://example.com/path?a=1&b=2' (& not encoded)

encodeURIComponent()

Encodes a URL component (like a query parameter value). It encodes everything including /, ?, & and =. Use this when encoding a value that goes inside a URL.

encodeURIComponent('hello world & foo=bar')
// → 'hello%20world%20%26%20foo%3Dbar'

// Correct way to build a URL with parameters:
var base = 'https://example.com/search';
var query = '?q=' + encodeURIComponent(userInput) + '&lang=en';
var url = base + query;

⚠️ Common mistake: Never use encodeURI() on individual query parameter values. It will not encode & and =, which will break your query string. Always use encodeURIComponent() for values.

Form Encoding — The + for Space

When HTML forms are submitted with method="GET" or Content-Type: application/x-www-form-urlencoded, they use a slightly different encoding: spaces are encoded as + instead of %20. Everything else follows percent encoding.

# Form encoded query string
name=John+Doe&message=Hello+World

# Percent encoded equivalent
name=John%20Doe&message=Hello%20World

When decoding form data, + must be decoded as a space. This is why you should use language-specific form parsing functions rather than raw URL decoding.

URL Encoding in Code

JavaScript

// Encode a full URL
encodeURI('https://example.com/path with spaces')

// Encode a component value
encodeURIComponent('foo & bar = baz')

// Decode
decodeURIComponent('hello%20world')

// Build query string properly
var params = new URLSearchParams({ q: 'hello world', page: 1 });
var url = 'https://example.com/search?' + params.toString();
// → 'https://example.com/search?q=hello+world&page=1'

Python

from urllib.parse import quote, unquote, urlencode, quote_plus

# Encode a path component (/ not encoded)
quote('/path/to/file with spaces')
# → '/path/to/file%20with%20spaces'

# Encode a query value (/ encoded)
quote('foo & bar', safe='')
# → 'foo%20%26%20bar'

# Form encoding (space → +)
quote_plus('hello world')
# → 'hello+world'

# Build query string
urlencode({'q': 'hello world', 'page': 1})
# → 'q=hello+world&page=1'

# Decode
unquote('hello%20world')

cURL

# curl handles encoding automatically with --data-urlencode
curl -G 'https://api.example.com/search' \
  --data-urlencode 'q=hello world & stuff'

# Or encode manually
curl 'https://api.example.com/search?q=hello%20world%20%26%20stuff'

Unicode Characters in URLs

Non-ASCII characters (like accented letters, Chinese characters, emoji) must be encoded as their UTF-8 byte sequence, with each byte percent-encoded:

# "café" in UTF-8
c  → 63
a  → 61
f  → 66
é  → C3 A9 (two bytes)

# Encoded
caf%C3%A9

# In a URL:
https://example.com/menu/caf%C3%A9

Modern browsers display the decoded Unicode in the address bar, but internally the URL uses percent encoding. This is part of the Internationalized Resource Identifiers (IRI) specification.

💡 Tip: Always use your language's built-in URL encoding functions rather than writing your own. Edge cases like Unicode, reserved characters, and form vs. URI encoding are easy to get wrong.

Decoding URLs

When you receive a URL or query string, you need to decode it before using the values in your application. Double-encoding (encoding an already-encoded URL) is a common bug to watch out for.

# Double-encoding bug example
original = 'hello world'
encoded = 'hello%20world'
double_encoded = 'hello%2520world'  # %25 = %, so %2520 = %20

# When you decode once:
unquote('hello%2520world')  → 'hello%20world'  # still encoded!
# When you decode again:
unquote('hello%20world')    → 'hello world'    # correct

Frequently Asked Questions

What does %20 mean in a URL?

%20 is the percent-encoded representation of a space character. The hex code for space in ASCII is 20 (decimal 32). So whenever you see %20 in a URL, it's just a space.

What is the difference between %20 and + for spaces?

Both represent spaces, but in different contexts. %20 is the standard percent encoding used in URL paths. + is used in form-encoded query strings (application/x-www-form-urlencoded). In a path like /hello+world, the + is a literal plus sign, not a space.

Why do I see %2F in some URLs?

%2F is an encoded forward slash /. It's used when a slash is part of a data value (not a path separator). For example, a file path like 2023/report.pdf would be encoded as 2023%2Freport.pdf when used as a query parameter value.

Should I encode the whole URL or just the parts?

You should encode individual components (query parameter values, path segments) and then assemble the URL. Never encode a fully-assembled URL — it will double-encode characters that are already correct.
