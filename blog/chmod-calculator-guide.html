What is chmod?

chmod stands for change mode and is the Linux command used to set file and directory permissions. Every file on a Linux system has three sets of permissions: one for the owner, one for the group, and one for everyone else (other).

Each set contains three permission types: read, write, and execute. The chmod command lets you control exactly who can do what with your files.

Understanding Octal Notation

The most common way to use chmod is with octal (base-8) numbers. Each permission is assigned a value:

ValuePermissionSymbol
4Readr
2Writew
1Executex
0No permission-

You add these values together for each user group. So 7 means read+write+execute (4+2+1), 6 means read+write (4+2), and 5 means read+execute (4+1).

How to Use chmod

Basic syntax

chmod [permissions] [file or directory]

Set permissions on a single file

# Make a script executable
chmod 755 deploy.sh

# Set web file permissions
chmod 644 index.html

# Restrict a private key
chmod 600 ~/.ssh/id_rsa

Recursive — apply to all files in a directory

# Set all files in /var/www to 644
chmod -R 644 /var/www/html

# Set directories to 755, files to 644 (common web server setup)
find /var/www/html -type d -exec chmod 755 {} \;
find /var/www/html -type f -exec chmod 644 {} \;

💡 Tip: For web servers, directories usually need 755 (execute allows traversal) while files need 644 (no execute needed for HTML/CSS/JS).

Symbolic Mode vs Octal Mode

Besides octal numbers, chmod also accepts symbolic notation using letters:

# Add execute permission for the owner
chmod u+x script.sh

# Remove write permission for group and others
chmod go-w file.txt

# Set exact permissions using equals
chmod u=rwx,g=rx,o=r file.txt

Symbolic mode is useful when you want to add or remove a specific permission without knowing the current state. Octal is better when you want to set an exact state.

chmod for SSH Keys

SSH is very strict about key file permissions. If they are too open, SSH will refuse to use them:

# Private key — owner read/write only
chmod 600 ~/.ssh/id_rsa

# Public key — owner read/write, others read
chmod 644 ~/.ssh/id_rsa.pub

# authorized_keys file
chmod 600 ~/.ssh/authorized_keys

# .ssh directory itself
chmod 700 ~/.ssh

⚠️ Warning: Never set SSH private keys to 777 or 666. SSH will reject them with "Permissions too open" error.

chmod for Web Servers (Nginx / Apache)

The standard permission setup for web servers running as www-data:

# Ownership — give web server read access
chown -R www-data:www-data /var/www/html

# Directories need execute (traverse) permission
find /var/www/html -type d -exec chmod 755 {} \;

# Files just need read
find /var/www/html -type f -exec chmod 644 {} \;

# If your app needs to write files (uploads etc)
chmod 775 /var/www/html/uploads

Viewing Current Permissions

# List files with permissions
ls -la

# Example output:
# -rw-r--r-- 1 user group 1234 Mar 14 index.html
# drwxr-xr-x 2 user group 4096 Mar 14 uploads/

The first character is - for file or d for directory. The next 9 characters are three groups of rwx for owner, group and other.

Frequently Asked Questions

What is chmod 777?

chmod 777 gives read, write and execute permission to everyone — owner, group and other. This is rarely a good idea in production as it means any user on the system can modify the file. Only use it temporarily for debugging.

What is chmod 755?

chmod 755 gives the owner full permissions (rwx) and everyone else read and execute (r-x). This is the standard permission for web directories and executable scripts — others can read and run but not modify.

What is chmod 644?

chmod 644 gives the owner read and write (rw-) and everyone else read only (r--). This is the standard for web files like HTML, CSS and JavaScript — the web server can read them but not modify them.

Does chmod work recursively?

Yes, use the -R flag: chmod -R 755 /var/www. Be careful as this applies the same permission to both files and directories, which is usually not what you want. Use find with -type f and -type d to set different permissions for files and directories.
