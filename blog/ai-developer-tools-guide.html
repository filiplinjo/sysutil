Cron Expression Builder

Cron syntax is powerful but unintuitive. The five-field format (minute hour dom month dow) is easy to get wrong, especially for complex schedules involving weekdays, multiple times, or ranges. The AI Cron Builder lets you skip the syntax entirely.

How to use it

Go to sysutil.dev/ai, select Cron Builder, and type your schedule in plain English. Press Enter or click Generate.

Example
Input: "every weekday at 9am"
Expression 0 9 * * 1-5 Breakdown minute: 0 — on the hour
hour: 9 — 9am
dom: * — every day
month: * — every month
dow: 1-5 — Monday through Friday
More examples
"every 15 minutes" → */15 * * * *
"every Sunday at 2am" → 0 2 * * 0
"first day of every month" → 0 0 1 * *
"every hour between 8am–6pm" → 0 8-18 * * *
"twice a day at noon and midnight" → 0 0,12 * * *

💡 Once you have the expression, copy it directly into the Cron tool to see a human-readable description and next run times.

Regex Pattern Builder

Regular expressions are one of the most useful tools in a developer's arsenal — and one of the hardest to write correctly from memory. The AI Regex Builder takes a plain English description and returns a pattern with a breakdown of every component.

How to use it

Select Regex Builder, describe what you want to match, and click Generate. The result includes the pattern, recommended flags, a token-by-token breakdown, and example strings that match.

Example
Input: "IPv4 addresses"
Pattern \b(?:(?:25[0-5]|2[0-4]\d|[01]?\d\d?)\.){3}(?:25[0-5]|2[0-4]\d|[01]?\d\d?)\b
Examples that match 192.168.1.1 · 10.0.0.255 · 172.16.254.1

Good prompts for the regex builder

The more specific you are, the better the pattern. Compare these:

❌ "email" → basic pattern, may miss edge cases
✅ "email addresses, allow plus addressing like user+tag@domain.com"

❌ "URL" → overly broad
✅ "https URLs only, with optional path and query string, no fragments"

❌ "phone number" → ambiguous
✅ "UK mobile numbers starting with 07, with or without spaces"

💡 Copy the generated pattern directly into the Regex Tester to test it against your own strings immediately.

Linux Command Explainer

Shell pipelines can be intimidating. A command like find /var/log -name "*.log" -mtime +30 -exec gzip {} \; does something specific and important — but if you didn't write it, it takes real effort to decode. The Command Explainer breaks down any command token by token.

How to use it

Select Command Explainer, paste any shell command or pipeline, and click Explain. The AI returns a one-sentence summary and a per-token breakdown covering commands, flags, arguments, pipes, and redirects.

Example
Input: ps aux --sort=-%cpu | head -10
Summary Show the 10 processes currently using the most CPU.
Breakdown ps — list running processes
aux — show all processes for all users in BSD format
--sort=-%cpu — sort by CPU usage, descending
| — pipe output to the next command
head -10 — show only the first 10 lines

⚠ The explainer flags dangerous commands. If you paste something like rm -rf / or dd if=/dev/zero of=/dev/sda, it will show a clear warning before you run it.

Error Message Decoder

Error messages and stack traces are often cryptic — especially for systems you didn't build, languages you're not fluent in, or infrastructure errors that don't tell you what actually went wrong. The Error Decoder takes any error and returns the likely cause plus concrete steps to fix it.

How to use it

Select Error Decoder, paste your error message or stack trace, and click Decode. You get the error type, what caused it, numbered fix steps, and a prevention tip.

Example
Input: ECONNREFUSED 127.0.0.1:5432
Cause Your application tried to connect to PostgreSQL on port 5432, but nothing is listening there. Either PostgreSQL isn't running, or it's bound to a different address.
Fix steps 1. Check if PostgreSQL is running: systemctl status postgresql
2. Start it if not: systemctl start postgresql
3. Verify it's listening: ss -tlnp | grep 5432
4. Check your app's DB_HOST config matches where Postgres is actually bound

What kinds of errors work well

The decoder handles a wide range of errors including:

  • Node.js / Python / Go / Java stack traces
  • Linux system errors (ENOENT, EPERM, EACCES, etc.)
  • Database errors (PostgreSQL, MySQL, Redis)
  • Docker and Kubernetes errors (OOMKilled, CrashLoopBackOff, ImagePullBackOff)
  • SSL/TLS certificate errors
  • HTTP errors and nginx/Apache logs
  • Git errors

Tips for Better Results

Be specific

Vague prompts get vague results. Instead of "match URLs", try "match https URLs with optional query strings but no hash fragments". Instead of "explain this error", paste the full stack trace rather than just the first line.

Iterate

If the first result isn't right, refine your description. Add constraints ("but not private IP ranges"), clarify the context ("in a Python f-string"), or specify the format ("for use in JavaScript with the global flag").

Validate cron expressions

AI-generated cron expressions are usually correct but always verify them in the Cron tool before deploying to production. Edge cases like "last day of month" or "every weekday in February" can trip up the model.

Test regex patterns

Generated regex patterns should be tested against real inputs before use. Copy the pattern into the Regex Tester and run it against both valid and invalid examples to confirm it behaves as expected.
