What Is an SSL Certificate?

An SSL (Secure Sockets Layer) certificate — more accurately called a TLS certificate today — is a digital file that authenticates a website's identity and enables encrypted connections. When you visit a site with https://, your browser verifies the site's certificate before establishing a secure connection.

Certificates are issued by Certificate Authorities (CAs) — trusted organizations that verify identity before issuing a cert. Your browser comes with a list of trusted root CAs built in.

What a Certificate Contains

FieldDescription
SubjectThe domain or organization the cert belongs to
IssuerThe Certificate Authority that signed it
Valid From / ToThe certificate's active date range
SANsSubject Alternative Names — all domains covered
Serial NumberUnique ID assigned by the CA
FingerprintHash of the certificate — used to verify authenticity

Certificate Types

There are three validation levels, each with a different level of identity verification:

TypeValidationTypical UseCost
DVDomain onlyPersonal sites, blogsFree (Let's Encrypt)
OVOrganization verifiedBusiness websitesPaid
EVExtended — full legal entity checkBanks, enterpriseMore expensive

For most websites, a free DV certificate from Let's Encrypt is perfectly adequate. The padlock looks identical to end users regardless of certificate type.

How to Check an SSL Certificate

Using the SSL Checker tool

Visit sysutil.dev/ssl and enter any domain. You'll instantly see:

  • Valid / expiring soon / expired status
  • Exact expiry date and days remaining
  • Who issued the certificate
  • All domains covered (SANs)
  • Serial number and fingerprint

In your browser

Click the padlock icon in the address bar of any HTTPS site. Then click "Connection is secure" → "Certificate is valid" to see full certificate details including the expiry date and issuer.

From the command line (openssl)

# View full certificate details
echo | openssl s_client -servername example.com \
  -connect example.com:443 2>/dev/null | openssl x509 -noout -text

# Just the expiry dates
echo | openssl s_client -servername example.com \
  -connect example.com:443 2>/dev/null | openssl x509 -noout -dates

# Days until expiry (number only)
echo | openssl s_client -servername example.com \
  -connect example.com:443 2>/dev/null | openssl x509 -noout -enddate \
  | cut -d= -f2 | xargs -I{} date -d "{}" +%s \
  | xargs -I{} sh -c 'echo $(( ({} - $(date +%s)) / 86400 ))'

💡 Tip: Set up a cron job or monitoring alert to notify you 30 days before your certificate expires. A lapsed SSL cert will break your site for all visitors.

Understanding the Certificate Chain

Certificates work in a chain of trust. Your site's certificate is signed by an intermediate CA, which is in turn signed by a root CA that browsers trust.

Root CA (built into browser)
  └── Intermediate CA (issued by root)
        └── Your site certificate (issued by intermediate)

If your server doesn't send the intermediate certificate, some clients will show a warning even though the cert itself is valid. This is one of the most common SSL configuration mistakes.

Check your chain with openssl

# Verify the full chain
openssl s_client -servername example.com \
  -connect example.com:443 -showcerts 2>/dev/null \
  | grep -E "^(subject|issuer)"

Wildcard Certificates and SANs

A single certificate can cover multiple domains using Subject Alternative Names (SANs):

  • Wildcard — *.example.com covers all subdomains (api.example.com, www.example.com, etc.) but not the root domain or sub-subdomains
  • Multi-domain SAN — lists specific domains explicitly, can cover completely different domains
  • Let's Encrypt — issues free certs with up to 100 SANs per certificate

💡 Tip: Always include both example.com and www.example.com in your certificate's SANs. A cert for only one won't work for the other.

Automating SSL with Let's Encrypt

Let's Encrypt provides free, automatically-renewing certificates. On most Linux servers, Certbot handles everything:

# Install certbot (Ubuntu/Debian)
apt install certbot python3-certbot-nginx

# Issue a certificate for your domain
certbot --nginx -d example.com -d www.example.com

# Auto-renewal runs via cron or systemd timer
# Test renewal manually:
certbot renew --dry-run

Certificates are valid for 90 days. Certbot auto-renews when there are fewer than 30 days left. Check the renewal timer:

systemctl status certbot.timer

Monitoring Certificate Expiry

For production servers, proactive monitoring prevents expired cert outages:

Simple cron-based check

#!/bin/bash
# Check SSL expiry and alert if < 30 days
DOMAIN="example.com"
EXPIRY=$(echo | openssl s_client -servername $DOMAIN \
  -connect $DOMAIN:443 2>/dev/null \
  | openssl x509 -noout -enddate | cut -d= -f2)
EXPIRY_EPOCH=$(date -d "$EXPIRY" +%s)
NOW_EPOCH=$(date +%s)
DAYS=$(( ($EXPIRY_EPOCH - $NOW_EPOCH) / 86400 ))

if [ $DAYS -lt 30 ]; then
  echo "WARNING: $DOMAIN SSL cert expires in $DAYS days" | mail -s "SSL Alert" admin@example.com
fi

⚠️ Note: An expired certificate will immediately break your site for all visitors. Most browsers show a full-page warning with no easy override, causing nearly 100% bounce rate.

HTTP Strict Transport Security (HSTS)

HSTS tells browsers to always use HTTPS for your domain — even if the user types http://. Add this header to your server response:

# Nginx
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;

# Apache
Header always set Strict-Transport-Security "max-age=31536000; includeSubDomains"

Once HSTS is active, browsers cache this directive. Be careful: if you let your certificate expire while HSTS is active, users won't be able to bypass the warning.
