What Is DNS?

DNS — the Domain Name System — is often called the phone book of the internet. When you type example.com into your browser, your computer doesn't know where that is. It needs a numeric IP address like 93.184.216.34 to actually reach the server. DNS is the system that performs this translation.

Every device on the internet has an IP address. Domain names exist because humans are much better at remembering words than strings of numbers. DNS bridges that gap — it's a globally distributed database that maps human-readable names to machine-readable addresses.

DNS was designed in the early 1980s and has scaled to handle hundreds of billions of queries per day across the entire internet, all while remaining fast enough that lookups typically complete in milliseconds.

How DNS Resolution Works Step by Step

When your browser needs to resolve a domain name, a multi-step process kicks off behind the scenes. Most of the time this takes under 50 milliseconds — so fast you never notice it.

The four players

ComponentRole
Recursive ResolverYour ISP's (or chosen public) DNS server — does the legwork on your behalf
Root Name Servers13 root server clusters worldwide — the top of the DNS hierarchy
TLD Name ServersHandle top-level domains like .com, .org, .net — know which authoritative server to ask next
Authoritative Name ServerThe final answer — holds the actual DNS records for the domain

The resolution sequence

1. You type "example.com" in your browser.

2. Browser checks its own cache → not found.
   OS checks /etc/hosts → not found.
   OS asks the Recursive Resolver (e.g. 8.8.8.8).

3. Recursive Resolver checks its cache → not found.
   Asks a Root Name Server: "Who handles .com?"

4. Root Name Server replies:
   "Ask the .com TLD server at 192.5.6.30"

5. Recursive Resolver asks the .com TLD server:
   "Who handles example.com?"

6. TLD server replies:
   "Ask the authoritative server at ns1.example.com"

7. Recursive Resolver asks ns1.example.com:
   "What is the A record for example.com?"

8. Authoritative server replies: "93.184.216.34, TTL 3600"

9. Recursive Resolver caches the result for 3600 seconds,
   then returns 93.184.216.34 to your browser.

10. Your browser connects to 93.184.216.34 — the page loads.

Tip: Steps 3–8 only happen on a cache miss. On subsequent lookups within the TTL window, the resolver returns the cached answer immediately — which is why DNS feels near-instant for popular domains.

A record — the workhorse

The A record is what most people mean when they say "DNS record." It's a direct mapping from a hostname to an IPv4 address. A domain can have multiple A records pointing to different IPs for load balancing or redundancy — resolvers typically return all of them and clients rotate through.

CNAME — aliases and subdomains

CNAMEs are useful for subdomains that should track another name. For example, www.example.com might CNAME to example.com, so you only need to update one A record when your IP changes. Important constraint: you cannot use a CNAME on an apex/bare domain (example.com itself) — only on subdomains.

MX — email routing

MX records tell the internet where to deliver email for your domain. Each MX record has a priority number — lower numbers are tried first. If the lowest-priority server is unreachable, the sender tries the next one.

# Example MX setup with primary and backup
example.com  MX  10  mail1.example.com   ← primary (tried first)
example.com  MX  20  mail2.example.com   ← backup

TXT — verification and email security

TXT records are used for a wide variety of purposes, all relying on the ability to store readable text in DNS:

  • SPF — lists servers authorized to send email from your domain
  • DKIM — stores the public key used to verify email signatures
  • DMARC — defines policy for how receivers handle SPF/DKIM failures
  • Domain verification — Google, GitHub, and others ask you to add a TXT record to prove domain ownership

How to Do a DNS Lookup

Using the online DNS Lookup tool

The fastest way is to use sysutil.dev/dns. Enter any domain name, select the record type you want (A, MX, TXT, etc.), and hit lookup. You'll immediately see all matching records along with their TTL values — no installation required, works from any device.

Tip: Use the online tool when you need to see what the public internet sees for a domain — especially useful during DNS propagation when your local resolver's cache may be stale.

Using nslookup

nslookup is available on Windows, macOS, and Linux without installing anything extra. It's the quickest command-line option for basic lookups.

# Basic A record lookup
nslookup example.com

# Look up a specific record type
nslookup -type=MX example.com
nslookup -type=TXT example.com
nslookup -type=NS example.com
nslookup -type=AAAA example.com

# Query a specific DNS server (bypasses your default resolver)
nslookup example.com 8.8.8.8
nslookup example.com 1.1.1.1

# Reverse DNS lookup (IP to hostname)
nslookup 93.184.216.34

# Interactive mode — run multiple queries
nslookup
> set type=MX
> example.com
> exit

Using dig (recommended for deeper inspection)

dig (Domain Information Groper) gives you more detail than nslookup and is the preferred tool for diagnosing DNS issues. It's available by default on macOS and Linux; on Windows, install it via BIND or use WSL.

# Basic A record lookup
dig example.com

# Specific record type
dig example.com MX
dig example.com TXT
dig example.com NS
dig example.com AAAA

# Short output — just the answer
dig example.com A +short

# Query a specific DNS server
dig @8.8.8.8 example.com A
dig @1.1.1.1 example.com MX

# Trace the full resolution path (root → TLD → authoritative)
dig +trace example.com

# Reverse DNS lookup
dig -x 93.184.216.34

# Show only the TTL
dig example.com A +noall +answer

# Check DNSSEC
dig example.com +dnssec
# Example dig output — reading the ANSWER SECTION
$ dig example.com A +noall +answer

;; ANSWER SECTION:
example.com.      3600    IN    A    93.184.216.34
#   ^hostname     ^TTL         ^type  ^value

Tip: dig +trace is invaluable for debugging. It shows every step of the resolution chain, which makes it obvious if a delegation is broken or if the wrong nameserver is answering.

TTL Explained — Time To Live

Every DNS record has a TTL (Time To Live) value measured in seconds. TTL controls how long a resolver or cache is allowed to store a record before discarding it and fetching a fresh copy from the authoritative server.

TTL ValueDurationTypical Use Case
601 minuteFailover setups, active migrations
3005 minutesLow-propagation-delay scenarios
36001 hourStandard default for most records
8640024 hoursStable records that rarely change (NS, SOA)

Why TTL matters

When you update a DNS record, the old value stays cached in resolvers around the world until their TTL expires. If your A record had a TTL of 86400, some users could see the old IP for up to 24 hours after you make a change. This is DNS propagation.

Warning: If you're planning a server migration, lower your TTL to 300 (5 minutes) at least 24–48 hours before the change. This minimizes the propagation window. Remember to raise it back after migration is complete.

DNS Propagation — Why Records Don't Update Instantly

When you update a DNS record at your registrar or DNS host, the change doesn't appear everywhere at the same time. This delay is called DNS propagation, and it's one of the most commonly misunderstood aspects of DNS.

Why propagation takes time

There is no central "push" mechanism in DNS. Instead, every resolver worldwide caches records independently and only fetches fresh data when the TTL expires. The timeline looks like this:

T=0:00   You update the A record for example.com at your DNS host.
         The authoritative server now returns the new IP.

T=0:01   Users whose resolver hadn't cached the record get the new IP.

T+1hr    Resolvers that cached a TTL=3600 record start getting the new IP.

T+24hr   Resolvers that cached a TTL=86400 record finally update.
         Most of the world is now seeing the new record.

Checking propagation from different locations

To see what different resolvers around the world are returning, use the online DNS tool at sysutil.dev/dns or compare results by querying multiple public DNS servers directly:

# Compare what Google DNS vs Cloudflare vs your ISP sees
dig @8.8.8.8   example.com A +short
dig @1.1.1.1   example.com A +short
dig @9.9.9.9   example.com A +short
dig            example.com A +short    # your default resolver

Tip: If all three public resolvers return the new IP but some of your users still see the old one, their ISP's resolver is the one that hasn't expired its cache yet. There's nothing you can do except wait for their TTL to expire.

Common DNS Issues and How to Diagnose Them

Domain resolves to the wrong IP

After a server migration, some users reach the old server. Check that the A record at your DNS host shows the new IP, then verify different resolvers are returning it. The likely cause is stale cache — wait for the old TTL to expire.

# Check what the authoritative server says (bypasses all caches)
dig @ns1.yourdns.com example.com A +short

Email not delivering — MX record problems

If email to your domain is bouncing, your MX records may be missing, incorrect, or pointing to a non-existent hostname. Check both the MX record itself and that the target hostname has a valid A record.

# Check MX records
dig example.com MX +short

# Verify the mail server hostname resolves
dig mail.example.com A +short

Warning: Never point an MX record at a CNAME. The DNS specification (RFC 2181) prohibits this. Always point MX records at hostnames that have their own A records.

Subdomain not resolving

If www.example.com or api.example.com isn't resolving, check that an A record or CNAME exists for that exact subdomain in your DNS zone. A common mistake is adding the record as www.example.com. (with trailing dot causing a double-domain issue) or omitting it entirely.

dig www.example.com A +short
dig api.example.com A +short

SPF / DKIM failing — TXT record issues

Email authentication failures usually mean your SPF, DKIM, or DMARC TXT records are missing or malformed. Verify the TXT records exist and contain the exact strings your mail provider specifies.

# Check SPF
dig example.com TXT +short | grep spf

# Check DMARC
dig _dmarc.example.com TXT +short

# Check DKIM (replace "selector" with your actual DKIM selector)
dig selector._domainkey.example.com TXT +short

NXDOMAIN — domain doesn't exist

An NXDOMAIN response means the DNS system has no records for that name at all. This could mean the domain has expired, was never registered, or the record was accidentally deleted from the zone.

# dig shows NXDOMAIN in the status field
dig nonexistent.example.com A
# ;; ->>HEADER<<- opcode: QUERY, status: NXDOMAIN
